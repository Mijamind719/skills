# LLM Wiki - 用 LLM 持续积累你的知识

> "与其每次查询时从原始文档重新检索，不如让 LLM 逐步构建和维护一个持久的 Wiki。"

## 核心观点

我以前也用过 RAG、NotebookLM、ChatGPT 文件上传——每次都是把文档丢进去，问问题时再让 AI 从零开始找。问题是：**每次都在重新发现知识**。问一个需要综合 5 份文档的问题，AI 得每次都去拼凑相关内容。知识没有积累，只有消耗。

这个 pattern 不一样。LLM 会在你和原始文档之间建立一个**持续更新的 Wiki**：
- 你放入新文档，LLM 会阅读、提取、整合到现有 Wiki
- 更新实体页、修订摘要、标注矛盾
- 知识编译一次，然后**持续保持最新**，而不是每次查询重新推导

LLM 做的是繁琐的记账工作：总结、交叉引用、归档、维护。
人做的是思考：筛选来源、引导分析、提出好问题。

## 为什么推荐这个方法

这是 Karpathy 提到的 "No Map Yes Context" 理念的实践：
- 一个巨大的 AGENTS.md = 一本 1000 页的说明书，没人有耐心读
- Context 是稀缺资源，放太多指令就会挤掉实际的代码和任务
- 最好的办法是直接给 AI 需要的上下文，而不是让它去翻手册

我们还发现：**直接把相关信息塞进 context 比 RAG 效果更好**。因为 Wiki 已经帮你整理好了关联知识，AI 拿到的是结构化、交叉引用的内容，而不是零散的检索片段。

## 快速开始

### 第一步：初始化 Wiki

告诉我你想创建什么样的知识库：
- 个人成长/健康/心理学？
- 研究课题？
- 项目文档？
- 团队知识库？

我会帮你创建目录结构：

```
your-wiki/
├── CLAUDE.md            # 维护规范，固化给 AI 的指令
├── raw/sources/         # 原始文档，只管丢这里
├── wiki/
│   ├── index.md         # 内容索引
│   ├── log.md           # 操作日志
│   ├── entities/        # 实体页（人、工具、项目...）
│   ├── concepts/        # 概念页（方法论、理论...）
│   └── source-summaries/# 资料摘要页
```

### 第二步：放入文档

把文件拖进 `raw/sources/`，然后告诉我：
> "帮我把 xxx.md 导入 wiki"

LLM 会：
1. 读取文档，提取关键信息
2. 创建摘要页
3. 更新/创建相关的实体页和概念页
4. 自动建立交叉引用
5. 更新 index.md 和 log.md

### 第三步：查询和使用

要写材料、分析问题、讨论观点时，直接问：
> "我最近看了哪些关于 AI Agent 的文章？"
> "帮我比较一下 Cursor 和 Windsurf 的区别"
> "总结一下我对这个项目的所有笔记"

## 日常维护

### 放入新文档后

```bash
/llm-wiki ingest raw/your_new_file.md
```

或者直接告诉我文件名，我会自动处理。

### 定期 Lint（建议每月或每季度一次）

```bash
/llm-wiki lint
```

我会检查：
- 页面间的矛盾点
- 过时的内容（原始文档很新但 wiki 很久没更新）
- 孤立页面（没有被任何地方引用）
- 缺失的交叉引用
- CLAUDE.md 是否需要更新

## Q&A

**Q: 一定要用 Claude Code 吗？**
A: 不一定。这个 pattern 适用于任何 LLM Agent（OpenAI Codex、OpenCode、Pi 等）。Skill 文件只是把工作流固化下来，你的 Agent 能执行就行。

**Q: 文件太多会不会很乱？**
A: raw/sources/ 是扁平的，不需要你手动分类。LLM 会通过 index.md 管理一切。你需要做的只是"把文件丢进去"。

**Q: 我的笔记在 Obsidian 里怎么办？**
A: 可以直接用 Obsidian 作为 IDE 打开 wiki 目录。LLM 写 Wiki，你用 Obsidian 浏览，双向链接和图谱视图都能用。

**Q: 这个和直接用 ChatGPT 上传文件有什么区别？**
A: 最大区别是**持久性**。ChatGPT 上传文件，问完就忘了。下次问同样的问题，AI 又要从头开始。但 Wiki 会记住——交叉引用已经在那里，矛盾已经被标注，综合分析已经完成。知识是会积累的。
